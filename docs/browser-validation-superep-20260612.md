## 발견 이슈

### A. [P1 후보] diagnose 폼 섹션1 상태 소실 — 세션 중 3회 재현

- 증상: 불량유형 칩 + 불량 설명(섹션1)이 입력 후 수 분 내 **소리 없이 리셋**. 수지/셋팅(섹션2·3)은 유지됨.
- 실해: 케이스4 1차 시도에서 칩=웰드라인·설명=파단 입력이 소실된 채 제출 → **플래시로 오진** (설명 없으니 LCP commonDefects prior로 확신 진단). 사용자 크레딧 1개와 신뢰를 동시에 잃는 시나리오.
- 부수 증상: 같은 타이밍에 제출 클릭이 무반응(크레딧 미차감, 결과 없음)인 경우 2회.
- 원인 후보(추정): 주기적 re-render(Supabase 토큰 리프레시/onAuthStateChange?)가 섹션1 상태를 초기화 / 칩·설명 상태가 다른 컴포넌트 키에 묶여 remount. 코드 조사 필요.
- 영향: 현장 사용자는 폼 작성에 수 분 걸림(사진 찍고 셋팅값 입력) — 재현 조건과 정확히 일치.

### B. [v1.4 보강] phase 동적판정 과교정

- 케이스3: 미성형(충전 단계 불량)인데 1순위 원인이 금형온도라서 phase가 **"냉각(Cooling) 불량"** 으로 표기.
- v1.3 규칙("phase = 채택 원인의 발생 단계")의 부작용. 올바른 기준은 **"결함 메커니즘이 작동/형성되는 단계"** — 미성형은 충전 중 응고로 미충전(filling), 과보압 크랙은 보압 중 응력 형성(packing). 원인이 금형온도(냉각 시스템)여도 형성 단계는 충전.
- 케이스5의 "냉각 불량"(후결정화)은 타당 — 결정화는 냉각 단계 메커니즘. 즉 규칙 문구만 정밀화하면 됨.

### C. [개선 검토] 무설명 제출 시 확신 과잉

- 케이스4 1차(설명·사진 없이 수지+셋팅만): 65% 확률로 플래시 확신 진단. 입력 빈약 시 "정보 부족 — 사진/설명 추가 시 정확도 향상" 같은 불확실성 신호 강화 검토 (이미 1차 추정 라벨은 있으나 본문 톤은 확신조).

### D. [관찰] 소소한 UX

- 진단 완료 후 헤더 크레딧 배지가 즉시 갱신 안 됨 (다음 페이지 로드에서 반영).
- 결과 패널 도착 시 자동 스크롤 없음 — 사용자가 스피너 위치에 있으면 결과 시작점을 놓칠 수 있음.
- (자동화 환경 한정 가능) 페이지가 document idle에 장시간 미도달하는 경우 1회 — next-pwa SW 캐싱 추정, 실사용 영향 미확인.

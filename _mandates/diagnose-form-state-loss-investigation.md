# Mandate: diagnose 폼 섹션1 상태 소실 — 조사 + 수정

> 단방향 Cowork→CC. 정본. 작성 2026-06-12.
> 근거: `docs/browser-validation-superep-20260612.md` 이슈 A (프로덕션 실측, 세션 중 3회 재현).

## 증상 (실측)

1. `/diagnose`에서 불량유형 칩 + 불량 설명(섹션1)을 입력하고 수 분 경과하면 **섹션1만 소리 없이 초기화**. 수지 select·강화재·셋팅값(섹션2·3)은 유지.
2. 같은 타이밍에 "AI 추정 시작" 클릭이 무반응인 경우 동반 (크레딧 미차감·스피너 없음·에러 표시 없음).
3. 실해 사례: 웰드라인+파단 설명이 소실된 채 제출되어 플래시로 오진 (사용자는 자기 입력이 사라진 걸 모름).

## 조사 포인트 (순서대로)

1. `app/diagnose/page.tsx`에서 칩 선택(defectType)·설명(defectDescription) 상태가 섹션2·3 상태와 **다른 생명주기**를 갖는 지점 — 컴포넌트 분리/key/조건부 렌더 여부.
2. `contexts/AuthContext.tsx`의 `onAuthStateChange`(TOKEN_REFRESHED 등) 발화 시 리렌더/리마운트 경로 — 토큰 리프레시 주기와 "수 분 후 소실" 타이밍 부합 여부. window focus 이벤트(`visibilitychange` 후 세션 재확인)도 동일 선상.
3. "최근 추정 N건" 배너(localStorage 읽기)가 마운트 후 늦게 나타나며 레이아웃/리마운트를 유발하는지.
4. 제출 무반응: 검증 실패 시(필수값 소실) 사용자에게 보이는 피드백이 있는지 — 없으면 그것도 버그.

## 수정 요구

- 근본 원인 수정 (리마운트 제거 또는 상태 끌어올리기).
- **방어선 추가**: 폼 전체 상태를 sessionStorage에 디바운스 저장 + 마운트 시 복원 (새로고침/리마운트 모두 커버). 진단 성공 시 클리어.
- 제출 검증 실패 시 명시적 토스트/메시지 (조용한 무반응 금지).
- 거버넌스·디자인 규율 준수 (토큰만, 17px+, 44px+).

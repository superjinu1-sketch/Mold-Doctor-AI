# Mandate: 트러블슈팅 일지 (진단 기록 보기) v0

> 단방향 Cowork→CC. 정본. 작성 2026-06-05.
> 철학(JINSIM): 진심으로 작업자를 돕는다 — 진단기가 아니라 작업자의 현장 노하우가 쌓이는 "오답노트+정비일지". 리텐션 + 데이터 플라이휠 + 성장도구.

## 목적
사용자가 **두 번째 방문부터 지난 진단 기록을 다시 보고**, 해결 과정을 남길 수 있게. v1·v2 로직은 나중이라도 **UI 그릇(자리)은 지금 판다.**

## 현황 (확인됨)
- `app/diagnose/page.tsx`: `diagnoseHistory` localStorage 저장 이미 작동(최근 20건, `{...data, timestamp, id, round}`, session_id 포함). `handleResolved`로 `resolved`/`resolvedAt` 저장도 일부 있음.
- **구멍: 저장된 기록을 다시 보는 화면이 없음.** /history 라우트 없음, result 복원 경로 없음(새로고침하면 결과 소실). 진입점에 기록 노출 없음.

## 구현 범위 (v0, 현장 검증 불필요 — 데모 전 가능)

### 1. 기록 보기 화면 (신규 `/history` 또는 진단 페이지 내 패널)
- localStorage `diagnoseHistory` 목록 카드: 날짜 / 불량유형(defect_type.ko) / 수지 / 심각도 배지 / 해결여부 배지(resolved).
- 카드 탭 → **상세 복원**: `setResult(record)` + `setSessionId(record.session_id ?? null)` (팔로업 연속성). 진단 결과 화면 그대로 재현.
- 빈 상태(첫 방문): "아직 기록이 없습니다. 첫 추정을 시작해보세요." (깔끔)
- 모바일 우선(디자인 5대 규율 준수: 토큰만, 17px+, 터치 44px+, 라이트 단일).

### 2. 진입점 "지난 기록" 노출 (두 번째 방문부터)
- /diagnose 상단(또는 홈)에 기록이 1건 이상이면: "최근 추정 N건" 요약 + "전체 기록 보기" 링크.
- 기록 0건이면 미노출(첫 방문 깔끔).

### 3. 해결 기록 보강 (handleResolved 확장, v0)
- 현재 `resolved` 불린 → **상태 3종**(해결 / 부분 해결 / 미해결) + **"어떻게 해결했나" 짧은 메모**(선택 입력, 마찰 최소 — 미입력 허용).
- **before/after 셋팅 자동 캡처**: 진단 시 입력한 settings/advSettings를 record에 함께 저장(작업자가 또 안 적게). 이미 data에 없으면 저장 시점에 추가.
- 기록 상세에 "✅ 이렇게 해결함: [상태 + 메모 + 바뀐 셋팅]" 표시 → 본인이 나중에 참조.
- **해결된 부품 사진 업로드 (옵션)**: 해결 기록 시 첨부 가능(선택, 미업로드 허용). 기존 사진 업로드 UI(카메라/드래그/Ctrl+V) 재활용 + downscale 적용. **옵션 칸에 헬프텍스트 기재**: "해결된 부품 사진을 올리면 보고서에 불량 전/후가 함께 들어가 더 명확해집니다." (왜 올리면 좋은지 설명)

### 4. v1 그릇 미리 파기 (placeholder)
- 기록 상세에 "유사 사례" 섹션 자리: 지금은 비활성/"곧 제공"(같은 불량·수지 과거 기록을 보여줄 자리). 클릭 동작 없어도 됨, UI 슬롯만.
- **데이터 구조를 v1(유사검색)·v2(공유) 고려해 구조화 유지**: record에 {defectType, resin, settings(before), recommendations, resolved상태, 조치메모, after셋팅, timestamp} 일관 보관.

### 5. 해결 리포트 PDF export ("위에 보고용")
현장 작업자가 상사에게 보고하는 실무 페인 해결. 기존 진단결과 PDF(onSavePDF, jsPDF+html2canvas)와 **별개의 보고서 전용 레이아웃**.
- 기록 상세(특히 해결된 케이스)에 "리포트 내보내기(PDF)" 버튼.
- 리포트 구성(A4 인쇄용):
  - 헤더: "사출 불량 개선 보고서" + 날짜 + 작업자/업체명(선택 입력, 미입력 허용)
  - 불량 정보: 유형·수지·**불량 사진(before, 있으면)**
  - 추정 결과: 원인(우선순위)·권장 조정
  - **조치 내역: before→after 셋팅 비교 표** (해결기록 자동 캡처값 활용)
  - 해결 결과: 상태(해결/부분)·조치 메모 + **해결 부품 사진(after, 있으면) → 불량 사진과 나란히 전/후 비교**
  - 체크리스트
- 기존 jspdf/html2canvas 재활용. 단 화면 캡처보다 **보고서 전용 프린트 레이아웃**(가독성 — 데모 첫인상). 깨진 폰트·잘림 없게.
- 거버넌스: "추정/조정안" 표기, 제조사 브랜드명 0, 리포트 하단에 "AI 추정 기반 — 최종 판단은 현장 검증 필요" 명시(100% 정확 주장 금지).

## 저장소
- v0 = **localStorage** (이미 있음, 데모 충분 — 작업자 본인 폰 기준 유지). 새 의존성·비용 0.
- **이미지 용량 주의**: 사진(불량·해결)을 base64로 localStorage 저장 시 용량 한계(~5MB) 빨리 참 → **downscale 후 저장**(기존 `lib/downscale` 재활용), 기록당 이미지 수 제한, 한도 초과 시 오래된 기록의 이미지부터 정리(메타데이터는 유지). 용량 초과 예외(QuotaExceededError) 핸들링 필수.
- 주석으로 남길 것: "로그인 사용자는 추후 Supabase 동기화로 기기 무관 + 플라이휠 연료 = v1 시점. 지금은 localStorage." (베타-B에 diagnoses 테이블 흔적 있으나 v0는 미사용.)

## 검증
- `npm run build` + `npm run verify` 통과.
- 흐름: 진단 1건 → /history(또는 패널)에 카드 뜸 → 탭하면 결과 복원 + 팔로업 가능(session_id 복원) → 해결 상태+메모 저장 → 재방문 시 진입점에 "지난 기록" 노출.
- 회귀: 기존 진단·크레딧 차감·데모 흐름 영향 없음(순수 추가).

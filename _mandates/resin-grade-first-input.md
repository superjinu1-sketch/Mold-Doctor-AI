# Mandate: 수지 입력 그레이드 우선(Grade-first) 재설계 — 수기 입력 최소화

> 단방향 Cowork→CC. 정본. 작성 2026-06-12. 진우 지시.
> **선행: `diagnose-form-state-loss-investigation.md` 머지 후 실행** (같은 page.tsx 충돌 방지).
> 철학(JINSIM): 현장 작업자의 수기 입력을 최소화. 포대 라벨 사진 또는 그레이드명 하나로 수지 정보가 채워지는 흐름.

## 목표 UX (섹션 2 재배치)

```
[2] 수지 정보
  ┌─────────────────────────────────────────────┐
  │ 📷 수지 포대 라벨 찍어 자동 입력              │   ← 신규 (사출기 화면 OCR 버튼과 동일 패턴)
  └─────────────────────────────────────────────┘
  수지 그레이드명                                    ← 기존 하단 필드를 최상단으로
  [ 예: PA66-GF30, 케피탈 F20-03, Ultramid A3EG6 ]  [자동 입력]
  ─ 자동 채움 성공 시: 아래 필드 채워짐 + "그레이드 기준 자동 입력됨 — 확인 후 수정 가능" 배지(신뢰도 낮으면 warn 톤)
  ─ "그레이드명이 없나요? (파일럿·컴파운드 등) 직접 입력" 토글 → 수기 필드 펼침
  수지 종류* / 강화재·함량 / 난연 3종 / 수지 상세    ← 기존 필드 유지, 기본은 접힘.
                                                      자동 채움 후엔 펼쳐진 상태로 값 표시(수정 가능)
```

원칙: 자동 채움은 **제안**이다. 필드는 항상 수정 가능, 최종 확정은 사용자. 수지 종류(필수)가 비면 기존 검증 그대로 작동.

## 작업 1 — UI 재배치 (app/diagnose/page.tsx + messages/ko·en)

- `resinGrade` 입력을 섹션 2 최상단으로 이동 + [자동 입력] 버튼(type="button", 44px+).
- 수기 필드 그룹(수지 종류~수지 상세)을 접이식으로: 기본 접힘, ① 토글 클릭 ② 자동 채움 성공 ③ 기존 임시저장 값 존재 — 셋 중 하나면 펼침.
- 자동 채움 배지: confidence high/med → brand-tint 배지, low → warn 배지 "확인 필요". 시맨틱 토큰만.
- 샘플 데모(PA66 은줄) 스냅샷 비교 로직이 깨지지 않는지 확인 (필드 값 자체는 동일, 배치만 변경).

### 실패 상태 메시지 (필수 — 조용한 실패 금지)

| 상황 | 사용자 메시지 (warn 톤, 그레이드 입력 필드 아래 인라인) | 후속 동작 |
|------|------|------|
| 조회 결과 null (미상 그레이드) | "이 그레이드 정보를 확인하지 못했습니다. 그레이드명에 오타가 없는지 확인하거나, 아래에서 직접 입력해주세요." | 수기 필드 자동 펼침 + 그레이드 입력에 포커스 유지 |
| confidence: low | "자동 입력된 정보의 신뢰도가 낮습니다. 수지 종류와 함량을 확인해주세요." | 필드 채우되 warn 배지, 수기 필드 펼침 |
| 라벨 사진 인식 실패 | "사진에서 그레이드명을 읽지 못했습니다. 라벨이 선명하게 나오게 다시 찍거나, 그레이드명을 직접 입력해주세요." | 수기 경로 안내 |
| 네트워크/서버 오류 | "일시적인 오류입니다. 잠시 후 다시 시도해주세요." | 입력값 보존 (소실 금지) |
| rate limit 초과 | "오늘 자동 조회 한도를 초과했습니다. 직접 입력은 계속 가능합니다." | 수기 필드 펼침 |

- 메시지는 i18n 키로 (ko/en), 본문 17px, danger 아닌 warn 톤 (사용자 잘못이 아님).
- 어떤 실패에서도 **이미 입력된 다른 필드 값을 건드리지 않는다.**

## 작업 2 — 그레이드 해석 API: POST /api/resolve-grade (신규)

**파이프라인 (순서 고정, api-cost-guard 원칙 준수):**

1. **정규화**: trim, 대문자화, 공백/하이픈 정리 → `cache_key`.
2. **캐시 조회** (신규 Supabase 테이블 `grade_cache`): hit면 즉시 반환. **외부 호출은 캐시 미스에만.**
3. **패턴 파서 (무료, lib/grade-parser.ts 신규)**: 정규식 규칙으로 추출 —
   - 수지 계열: `PA66|PA6|PBT|PC/ABS|POM|PPS|LCP|...` (RESIN_KB 키 enum 매핑, 긴 키 우선 매칭 — PA66을 PA6으로 오인 금지)
   - 강화재: `GF(\d+)|G(\d+)(?=[A-Z]|$)|CF(\d+)|TD(\d+)|탈크|미네랄` → filler enum + 함량
   - 난연: `V-?0|V-?2|5VA|FR|난연` → UL94 등급
   - 수지 계열이 확정되면 LLM 생략하고 결과 반환 (`source:'pattern'`).
4. **LLM 폴백 (haiku)**: 패턴으로 수지 계열 미확정 시에만. 시스템 프롬프트 요지:
   - 출력 JSON 강제: `{ resinType: <RESIN_KB 키 enum 또는 null>, filler, fillerPct, flameRating, flameType, confidence: 'high'|'med'|'low', note }`
   - **모르면 null + low — 추측 환각 금지.** 실재 불확실한 그레이드를 아는 척 금지.
   - **대체 그레이드·타사 제품 추천 출력 금지** (거버넌스).
   - max_tokens 512, 응답 방어적 파싱(ocr-parse-hardening 패턴 재사용).
5. **캐시 저장**: TTL 90일 (그레이드 스펙은 사실상 불변), hit_count 증가 로그.

**보안/한도**: extract-settings 패턴 그대로 — Bearer 인증 + 일일 rate limit RPC(신규 `increment_resolve_count`, 한도 30회/일) + 429. 크레딧 차감 없음. API 키 서버 전용.

**마이그레이션**: `supabase/migrations/0004_grade_cache.sql` — grade_cache 테이블 + rate limit RPC. 생성 후 진우가 SQL Editor에서 Run (베타-B 패턴).

## 작업 3 — 포대 라벨 사진 OCR: POST /api/extract-grade (신규)

- 기존 extract-settings와 **분리된 신규 라우트** (프롬프트 상이, 기존 OCR 회귀 0).
- Claude vision (`claude-sonnet-4-6`, extract-settings와 동일 모델·인증·rate limit·8MB 제한): 포대/라벨/성적서 사진에서 `{ manufacturer, gradeName, resinFamily, fillerText, flameText }` 추출. 인쇄 라벨 우선, 손글씨 보조.
- 추출된 gradeName을 **작업 2 파이프라인으로 통과**시켜 최종 스펙 반환 (서버 내부에서 연결, 클라 왕복 1회).
- 클라: 수지 섹션 상단 📷 버튼 → 업로드 → 자동 채움 + 그레이드명 필드에 인식값 표시.

## 거버넌스 / 주의

- 제조사·브랜드명: 사용자 입력의 **식별·표시(echo)는 허용** (추천이 아님). 단 AI가 먼저 특정 브랜드를 권하는 출력은 금지. lint-banned.sh의 브랜드 룰과 충돌 없는지 확인 — 충돌 시 보고 (룰은 "추천" 맥락이므로 식별 필드는 예외 처리 필요할 수 있음).
- 자동 채움 문구는 "자동 입력(추정) — 확인 후 수정" 톤 유지. "100% 정확" 류 금지.
- 디자인 5대 규율: 토큰만, 17px+, 44px+, 모바일 우선 (이 섹션은 현장 모바일 사용 핵심 경로).

## 비용 (참고 — 무료 제공 여부는 진우 결정 사항)

- resolve-grade: 패턴 히트·캐시 히트 = ₩0. LLM 폴백 haiku ≈ ₩1~3/회, 캐시 후 반복 ₩0.
- extract-grade: vision 1장 ≈ ₩6~10. rate limit 30회/일이면 악용 상한 ~₩300/일/계정.
- 본 mandate는 둘 다 **크레딧 무차감(무료)** 으로 구현. 유료화 전환은 진우 결정 후 별도.
